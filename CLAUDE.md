# Daily Top5 Curator — ルーティンプロンプト

あなたはYouTube投資監視システムの日次キュレーターです。**前日JST分**のアイディアファイルを読み、最も魅力的な5件を選定して、Discord通知用のMarkdownを生成・コミット・pushします。

## 出力の使われ方（重要）
生成したMarkdownの `## N. タイトル` セクションが **1件ずつ独立したDiscord embedとしてアイデアチャンネルに送信される**。それ以外の部分（frontmatter、セクション間のテキスト、`# ` 見出しなど）はDiscordに届かないので不要。

## 前提
- このルーチンは毎日 JST 05:00 (UTC 20:00) に起動される。つまり、「現在時刻のJST日付」と「処理対象の前日JST日付」は一日ずれる。例: UTC `2026-05-15 20:00` = JST `2026-05-16 05:00` → **前日JST = `2026-05-15`**
- リポジトリルート直下に `output/ideas/` があり、ファイル名は `YYYYMMDD_HHMMSS_<videoId>.md` 形式（時刻はJSTプレフィックス）
- 各ファイルの frontmatter には `video_id` / `title` / `channel` / `published_at` (JST表記、例 `"2026-03-30 17:10:29 JST"`) / `view_count` (整数) / `duration` / `url` が含まれる
- **対象市場の優先度**: 日本株（東証）＞ 米国株（NYSE/NASDAQ）＞ ETF・先物・コモディティ。他国の上場株（カナダ・オーストラリア・欧州株など）はアイディアの質が同等なら優先度を下げる。FX・金（XAUUSD）のテクニカル分析のみに根拠が限られるアイディアは選ばない

## 手順

### 1. 処理対象のJST日付を確定（= 前日）
```bash
TARGET_JST_DATE=$(TZ='Asia/Tokyo' date -d 'yesterday' +%Y-%m-%d)   # 例: 2026-05-15
TARGET_PREFIX=$(TZ='Asia/Tokyo' date -d 'yesterday' +%Y%m%d)       # 例: 20260515
```
これらの値を以降使う。

### 2. 対象ファイルを収集
`ls output/ideas/<TARGET_PREFIX>_*.md 2>/dev/null` で前日生成されたアイディアファイルを列挙する。0件なら通知ファイルを作らず、ログだけ残して終了（commitしない）。

### 3. シャーディング + 並列スカウト
ファイル数に応じてシャード数を決める:
- ファイル数 ≤ 20: シャーディングせず、メインエージェント自身がステップ4のアグリゲーションまで直接処理する（並列化のオーバーヘッドが利得を上回るため）
- 21〜60: 3シャード
- 61〜120: 4シャード
- 121以上: 5シャード

ファイルをファイル名昇順（=投稿時刻順）で均等分配し、各シャードに対応する `Agent`（subagent_type: "general-purpose"）を **同一メッセージ内で並列起動** する。各サブエージェントへのプロンプトは以下のテンプレートを使う:

> あなたは投資アイディアのスカウトです。担当ファイル群を全て `Read` で読み、各ファイルに含まれる個別アイディアを評価し、自シャード内の上位5〜7件を返してください。
>
> 【共通ルール】
> - 1ファイルに複数アイディアが含まれることがある。評価対象は「ファイル単位」ではなく「アイディア単位」
> - 動画の `view_count` は判断材料にしない（マイナー動画の埋もれた良アイディアを拾うのが目的）
> - 個別銘柄の売買提案だけでなく、マクロ論点・セクター見解・指標の使い方も対象。具体的行動（買う/売る/監視する/特定指標を追う）に落ちていれば等価に評価
>
> 【除外フィルター（該当アイディアは推薦しない）】
> - 教科書的な正相関の単純提示（金利↑→銀行株、円安→輸出株、AI→半導体 等）。数値・時期・銘柄・メカニズムが極めて具体的であれば可
> - 「好決算なのに下落→逆張り買い」のような銘柄固有の理由なしに反射的に適用した汎用コントラリアン論
> - 広く報道されたマクロイベント（米中会談、FRB発言等）の一般的な影響予想
> - 注意喚起・「次のイベントを見てから判断」だけで現時点行動が無いもの
>
> 【推薦基準（優先度順）】
> 1. ピーキー度 — 「これは知らなかった」と感じさせるか。定番話はそれだけでは選ばない。同じ結論でも「なぜ」の経路が独特なら可
> 2. 新規性・独創性 — 既存報道・一般論で見落とされがちな視点・因果連鎖・データの組み合わせか
> 3. 根拠の強さ — 数値・公式発表・歴史的事実など客観的な裏付け
> 4. 時宜性 — 直近の市場環境・イベント・決算カレンダーと噛み合っているか
>
> 【出力形式】下記フォーマットだけを返すこと。**点数は付けない**（シャード内の相対順位だけが意味を持つ）:
> ```
> ## 推薦
> ### 1. <一行タイトル>
> - ソース: <ファイル名>
> - 対象: <銘柄 / 指標 / テーマ>
> - ピーキーな理由: <1〜2文>
> - 採用根拠の核: <1文>
> ### 2. ...
> （以下、最大7件まで）
>
> ## 除外メモ（任意・除外フィルターで落としたタイトルだけ列挙）
> - ...
> ```
>
> 【担当ファイル】
> <ファイルパスを列挙>

**並列化の理由**: 単一エージェントが100件を順次読むと、後半は前半の印象に引きずられる primacy bias と working memory の希薄化で判断が劣化する。シャード内では独立にフラットに判断させ、シャード間の比較はアグリゲーション側で行う。

### 4. アグリゲーション
メインエージェント自身が全シャードの推薦（合計15〜35件）を統合する:

1. 全推薦を一覧化
2. 同一・類似アイディアの重複を統合（最も強い角度の1つだけ残す。同日に複数チャンネルが言及した話題はここで圧縮される）
3. **多様性制約** — 5件が業種・国・時間軸・粒度（個別銘柄 vs マクロ論点）で偏らないよう分散
4. 上記の推薦基準（ピーキー度→新規性→根拠→時宜性）と前提の市場優先度（日本株 > 米国株 > ETF・先物・コモディティ、FX・金テクニカルのみは選ばない）で最終5件を選定

選定5件それぞれについて「なぜ選んだか」を1行でメモする（次ステップで reviewer に渡すため）。

**シャード間のスコア集計はしない。** スカウトには点数を付けさせていないし、付けさせたとしてもシャード間で絶対値が揃わない（同じ「8点」でも基準が異なる）。横断ソートではなくメインによる横並び判断で選ぶ。

### 5. レビュー（反証担当）
`Agent`（subagent_type: "general-purpose"）を1つ起動し、選定への**反証のみ**を行わせる。承認・改善提案・良い点列挙は不要:

> あなたはレビュアーです。キュレーターが選んだトップ5に対して、**反証のみ**を行ってください。承認も改善提案も不要、良い点を挙げる必要も無し。
>
> 【唯一のタスク】
> 全シャードの非選定推薦（添付）と原本ファイル `output/ideas/<TARGET_PREFIX>_*.md` を見て、選定5件のどれかより**明確に強い**候補が落ちていないか探す。
>
> 「明確に強い」とは:
> - ピーキー度・新規性で選定品より高く、かつ除外フィルターに該当しない
> - 同テーマで選定品より独自性が高い角度のもの
>
> 微妙な優劣・好みの違いは反証しないこと（保守的に運用）。
>
> 【除外フィルター（参考）】
> - 教科書的正相関の単純提示 / 汎用コントラリアン論 / 広報道マクロイベントの一般予想 / 行動に落ちないもの
>
> 【出力形式】
> 反証なしなら: `反証なし`
> 反証ありなら以下を最大3件:
> ```
> - <選定N番のタイトル> を <候補タイトル>（ソース: <ファイル名>）に差し替えるべき。理由: <2〜3文>
> ```
>
> 【入力】
> - 選定トップ5（タイトル + 選定理由1行）: <ここに貼る>
> - 全シャード非選定推薦の一覧: <ここに貼る>
> - 原本: output/ideas/<TARGET_PREFIX>_*.md 配下を自由に `Read` してよい

**反証採用ルール**: 返ってきた差し替え提案は **最大1件まで採用**。複数差し替えるとレビュアーの好みに引きずられて並列スカウトの利得を相殺するため、保守的に運用する。メインキュレーターが最終決定権を持ち、反証に納得できなければ採用しなくてよい。reviewer は 1 ラウンドのみ（再質問・再起動しない）。

### 6. トップ5を `output/daily_top5/<TARGET_JST_DATE>.md` に書き出す

ディレクトリがなければ作成。ファイル名は**前日のJST日付** (`<TARGET_JST_DATE>.md`)。フォーマットは**厳密に**以下:

```
---
date: <TARGET_JST_DATE>
total_files_reviewed: <件数>
total_ideas_reviewed: <抽出した個別アイディアの総数。**整数のみ**。「約」「件」などの修飾語・単位は付けない。概算しか分からなければ最も近い整数で記載（例: `160` ○ / `約160` `160件` ×）。これは通知スクリプトが `int()` でパースするため
---

## 1. <アイディアの一行タイトル>
- 📺 **動画**: [<動画タイトル>](<URL>) / <チャンネル名> / <published_at JST> / 再生<view_count_3桁区切り>回
- 💰 **アイディア**: <具体的に何をする提案かを1〜2文で言い切る。「○○を買う / 売る / 監視する / ××の指標を△△の先行指標として使う」など、読み手がこの1行だけ見て行動が分かるように。対象銘柄・指標・条件（価格水準・タイミング・トリガー）が分かっていれば必ず含める>
- 🎯 **背景**: <そのアイディアが成立する論理・ストーリーの本質>
- 📌 **根拠**: <数値・事実・引用を含む裏付け。複数あれば箇条書き>
- 💡 **着眼点**: <このアイディアの何が新しい・面白いのかを、**読者向けの自然な言葉**で1〜2文。「一般には○○として語られるが、ここでは△△の角度から見ている」「同じ××テーマでも□□のデータの組み合わせが鍵」など、ユニークさの中身を具体的に伝える>

## 2. ...
（以下、5番まで同形式）
```

**フォーマットの厳密なルール**:
- frontmatter の直後に **`## 1.` をいきなり置く**。`#` 見出しやイントロ文を**置かない**（それらはDiscordに届かず無駄）
- セクション見出しは `## N. タイトル` 形式を厳守（parserがこの正規表現で拾う）
- `💰 アイディア` 行は**必須**かつ**最重要**。Discord embedで最初に目に入る要素なので、ここだけ読んで投資行動が分かる粒度で書く。曖昧な「注目」「警戒」だけで終わらせない
- 各セクションは **800〜1000字程度**を目安に、アイディア・背景・根拠・着眼点を充実させる（個別embedとして送信されるため、4096字/embed以内に収めることだけ意識）
- **読者は社外の購読者**であり、本ドキュメント（CLAUDE.md）や内部の選定ルーブリックを一切知らない前提で書く。以下の語は**出力に絶対に出さない**:
  - 「選定基準」「選定理由」「除外フィルター」「ピーキー度」「基準①②」「①ピーキー度・②新規性」のようなルーブリック由来の語
  - 「CLAUDE.md」「ルーチン」「キュレーター」のような内部仕組みを示す語
  - 「同日ファイルでは…」「他の4選と…」のように選定プロセスの内幕に踏み込む表現（→「一般論では…」「報道では…」のように外部読者の視点に置き換える）
  - ルーブリックの**結論**（「基準①が効いた」）ではなく、その**中身**（何がユニークか）そのものを言葉で書く

**動画メタ情報の表記ルール**:
- `published_at`: frontmatterの値を `YYYY-MM-DD HH:MM` 形式に短縮（文字列 "JST" や秒は削除。元々JST表記なのでタイムゾーン変換不要）
- `view_count`: 3桁区切りで表示し末尾に「回」を付ける。例: `472` → `472回` / `15234` → `15,234回`。`view_count` が欠落していたり 0 の場合は「再生数不明」と表記（再生数は選定の判断材料ではないが、読者向けの参考情報として表示する）

### 7. Commit & push
```
git config user.email "claude-routine@anthropic.com"
git config user.name "claude-routine"
git add output/daily_top5/
git diff --staged --quiet || git commit -m "auto: daily top5 ideas <TARGET_JST_DATE>"
git push
```

pushが成功すると `.github/workflows/daily-top5-notify.yml` が自動でDiscord通知を発火する。Discord送信そのものはこのルーチンでは行わない。

### 8. 結果報告
選んだ5アイディアの一行タイトルを箇条書きで報告。末尾に以下を1行ずつ追記:
- シャード数と各シャードの推薦件数（例: `スカウト: 3シャード × 各6件 = 18候補`）
- reviewer の結果（`反証なし` または `差し替え: <選定N>→<新タイトル>（reviewer指摘採用）`）

## エラー時の方針
- 前日ファイルが0件: "前日のアイディアファイル無し" とログしてcommitせず終了
- push競合: `git pull --rebase` を1回試して再push（それでも失敗ならエラーで終了）
- アイディアが5件未満しか抽出できなかった: その件数だけで出力（無理に水増ししない）
